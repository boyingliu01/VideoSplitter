本仓库的错误处理采用“分层 + 多策略”模式，覆盖底层 I/O/子进程、中间层工具包、上层 Pipeline 编排以及 GUI 用户反馈四个层次，核心约定如下：

1. 自定义异常类型（ffmpeg-skill）
- ffmpeg-skill/__init__.py 定义 FFmpegError(Exception)，作为所有 FFmpeg 相关错误的统一出口。
- _check_ffmpeg/_run_command/get_video_info 等方法将 FileNotFoundError、subprocess.TimeoutExpired、json.JSONDecodeError 等具体异常包装为 FFmpegError，并附带可诊断的 stderr 片段或返回码信息。
- 调用方（如 video_splitter/splitter/cutter.py）通过动态 import 复用该异常类，在精确切割失败时 raise FFmpegError，从而让上层能区分“FFmpeg 执行失败”这一领域错误。

2. 子进程与外部依赖错误（video_splitter/extractor/audio.py）
- get_duration/extract 使用 subprocess.run 调用 ffprobe/ffmpeg，对 returncode != 0 的情况 raise RuntimeError，并截断 stderr 前 500 字符作为错误消息。
- precheck 方法对 librosa/numpy 缺失采取“降级可用”策略：has_librosa=False 时直接返回 is_ok=True 的警告消息，不中断流程；内部 try/except Exception 捕获任何解析/计算异常后同样以“Pre-check skipped: …”形式返回，保证健壮性。

3. LLM 章节检测的容错回退（video_splitter/analyzer/chapter.py）
- ChapterDetector.detect 包裹整个 LLM 调用链路，一旦任意异常发生，立即回退到 _uniform_split（按 max_segment_duration 均匀切分），确保“即使 LLM 不可用也能产出结果”。
- _call_llm 实现指数退避重试（sleep(2**attempt)），并在循环内 catch 全部异常继续下一次尝试；_parse_response 借助 json_repair.repair_json 做 JSON 修复，再校验时间戳范围与 start < end 约束，越界抛出 ValueError 触发外层回退。
- _llm_request 在 openai 未安装时显式 raise RuntimeError("openai package required for LLM API")，明确依赖缺失语义。

4. Pipeline 编排层的错误收集与传播（video_splitter/pipeline.py）
- Pipeline.run 将整个流程包在一个 try/except Exception 中，无论哪一步抛错，都会把 result["status"] 置为 "error"、result["error"] = str(e)，并通过 logger.error 记录堆栈，随后重新 raise，使 CLI 或调用方能同时拿到结构化结果与原始异常。
- dry_run 路径不进入 try 块，遇到预检查失败直接返回 {"status": "error", "message": msg}，体现“只读路径”与“写入路径”不同的错误返回约定。

5. GUI 层的错误呈现（gui/controllers/review_controller.py + gui/app.py）
- ReviewController 暴露 error = Signal(str)，save_correction/export_srt 中的 IO 异常被捕获后通过 self.error.emit(str(e)) 上报，而不是直接崩溃。
- MainWindow 在 _connect_signals 中将 ctrl.error 连接到 _on_controller_error，后者弹出 QMessageBox.warning 向用户展示错误；健康检查阶段的异常则降级为状态栏文本，避免阻塞启动。
- TranscribeWorker 通过 QThread 异步运行，其 error 信号由 app 层统一转为 QMessageBox 提示，保持 UI 线程安全。

6. 开发者应遵循的规则
- 对外暴露的 API 尽量使用领域异常（FFmpegError）或标准异常（FileNotFoundError/ValueError/RuntimeError），不要吞掉异常；仅在确需降级处使用宽泛 except Exception 并记录日志。
- 涉及外部命令（ffmpeg/ffprobe/openai）的操作必须带 timeout，并将非零返回码转换为有意义的异常消息。
- 对可选依赖（librosa/json_repair/openai）采用“try-import + has_xxx 标志”的惰性加载，缺失时提供降级行为而非直接报错。
- GUI 侧禁止在主线程中执行耗时 I/O，错误一律通过 Qt Signal 上报并由顶层窗口统一弹窗或更新状态栏。
