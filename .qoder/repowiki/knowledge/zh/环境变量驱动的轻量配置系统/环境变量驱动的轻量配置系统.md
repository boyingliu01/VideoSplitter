## 系统概述
本项目采用**纯 Python dataclass + 环境变量覆盖**的极简配置方案，没有引入 pydantic-settings、toml/yaml 文件加载或集中式配置中心。所有运行时参数通过 `SplitConfig` dataclass 定义默认值，再通过 `from_env()` 类方法以 `os.environ.get(...)` 形式按优先级覆盖。

## 核心文件与包
- `video_splitter/config.py` — 唯一配置入口，定义 `SplitConfig` dataclass 及 `from_env()` 覆盖逻辑
- `video_splitter/cli.py` — CLI 子命令统一调用 `SplitConfig.from_env()` 构建配置，再对部分字段做命令行参数二次覆盖
- `video_splitter/pipeline.py` — 流水线编排器，构造时接受可选 `SplitConfig`，缺省则回退到 `from_env()`
- `gui/controllers/*.py`、`gui/workers/*.py` — GUI 侧同样通过 `config or SplitConfig.from_env()` 注入配置
- `video_splitter/extractor/engines.py` — FunASR 引擎特有环境变量（`VIDEO_SPLITTER_FUNASR_*`）在此处读取

## 架构与约定
1. **单一数据源**：`SplitConfig` 是全局唯一配置对象，所有模块（CLI、GUI、Pipeline、Worker）都依赖它，不存在分散的 `settings.py` 或配置文件。
2. **三层覆盖顺序**（从低到高）：
   - dataclass 默认值（硬编码）
   - 环境变量（`from_env()` 中 `os.environ.get(key, default)`）
   - CLI 参数（`cli.py` 中对 `max_duration`、`model`、`cut_mode`、`resume` 等字段做二次赋值）
3. **环境变量命名规范**：
   - 通用前缀 `VIDEO_SPLITTER_`：`DEVICE`、`RESUME`、`ENGINE`
   - ASR 引擎专用前缀 `VIDEO_SPLITTER_FUNASR_*`：`MODEL_DIR`、`PUNC_MODEL`、`CHUNK_SECONDS`
   - LLM 兼容 OpenAI 协议：`OPENAI_API_BASE`、`OPENAI_API_KEY`，并额外支持 `WHALECLOUD_API_KEY`（后者优先覆盖前者）
4. **无持久化配置文件**：项目根目录未发现 `.env`、`config.yaml`、`config.toml` 等文件；部署时通过容器/进程的环境变量注入配置。
5. **测试友好**：测试用例使用 `unittest.mock.patch.dict(os.environ, {...}, clear=True)` 隔离环境，避免污染全局状态。

## 开发者应遵循的规则
- **新增配置项**：在 `SplitConfig` dataclass 中添加字段并提供合理默认值，然后在 `from_env()` 中增加对应的 `os.environ.get(...)` 覆盖逻辑，保持注释同步更新。
- **环境变量命名**：应用级变量统一使用 `VIDEO_SPLITTER_` 前缀，引擎/子系统级变量使用 `VIDEO_SPLITTER_<SUBSYSTEM>_` 前缀，避免命名冲突。
- **敏感信息**：API Key 等敏感字段仅通过环境变量注入，禁止写入代码或版本库中的任何配置文件。
- **GUI 与 CLI 一致性**：GUI 组件通过构造函数参数传入 `SplitConfig` 实例，不直接读环境变量；CLI 负责将 `argparse` 参数合并到配置对象后再下发给各组件。
- **向后兼容**：`from_env()` 中所有 `os.environ.get` 调用必须提供默认值，确保未设置环境变量时行为可预测。
- **不要绕过 `from_env()`**：除测试外，生产代码应始终通过 `SplitConfig.from_env()` 获取配置，以保证环境变量覆盖生效。
